How

四大核心环节

1

仓库验证 — ttfhw-verify-pro

模拟新手开发者 · 只看文档不猜测
  • 从 README 入口理解仓库结构，识别系统要求、构建命令、测试入口
  • 自动选择并启动匹配的容器环境 (Ubuntu 24.04 / openEuler 22.03)
  • 执行构建 → 单元测试 → 样例运行 → 静态检查 (pre-commit / lint-runner) → devcontainer 检测
  • 记录每一步的时间戳、命令、输出、错误和返回码
  • 遇到依赖缺失时主动安装，不改源码
2

报告归一化 — ttfhw-report-normalizer

5 种源格式 · 语义映射 · 多 Agent 并行
  • 自动检测 5 种源格式：Legacy / Stratovirt / openEuler / SHMEM / Manifest
  • 状态值标准化为 5 种：success · partial_success · failed · not_run · no_tests
  • 修复常见 JSON 损坏（中文引号 / 垃圾字符 / 损坏 URL）
  • 保留 v630 扩展字段：static_analysis / devcontainer / duration_breakdown
  • 大批量任务启动多个 AI Agent 并行归一化，27 个文件 9 个 Agent 分钟级完成
3

静态面板 — Next.js SSG

零服务端 · GitHub Pages 免费托管 · 48 个静态页面
  • Next.js 16 App Router，Static Site Generation (SSG) 构建
  • 首页：状态统计、饼图、柱状图、可筛选仓库表（按社区 / 状态 / 搜索）
  • 45 个仓库详情页：构建 / UT / 样例 / 环境 / 时间线 / 原始 JSON
  • Recharts 可视化：通过率饼图 + 耗时对比柱状图
  • 每次归一化后自动重建并推送到 GitHub Pages，无需运维
4

飞书刷新 — ttfhw-feishu-update

17 列批量写入 · Dry-run 预览 · User 身份鉴权
  • 从归一化 JSON 提取 17 列核心数据：验证结果 / 时长 / 构建 / UT / 样例 / pre-commit / devcontainer / 验证问题
  • 仓库名与表格行号智能匹配，自动构建批量更新操作
  • Dry-run 预览机制：先预览所有变更，用户确认后执行
  • 基于 lark-cli +batch-update，单次调用完成全部单元格写入

Architecture

关键架构决策

决策点选择理由
验证执行者 AI Agent 模拟新手开发者 只有不看文档假设才能发现文档的缺失和错误；人类专家会不自觉脑补
报告归一化 AI 逐文件语义映射 5 种格式差异大，正则 / 脚本无法覆盖所有变体；每个文件需要完整上下文理解
多文件处理 独立 Agent 并行归一化 Agent 间无共享状态，互不影响；27 个文件 9 Agent 并行，分钟级完成
状态标准化 5 种标准值 中文 / 英文 / 混合状态值统一为 success / partial_success / failed / not_run / no_tests
前端架构 Next.js SSG 静态导出 零服务端成本，GitHub Pages 免费托管；纯静态 HTML 无运行时依赖
容器策略 Ubuntu 24.04 / openEuler 22.03 根据仓库文档特征自动选择匹配的容器镜像，确保可复现
飞书集成 lark-cli + User OAuth 绕过 Bot 写权限限制；Dry-run 预览 + 用户确认双保险
数据版本 json-org (原始) + json (归一化) 双目录结构保留原始数据和归一化数据，支持回溯和重新归一化
