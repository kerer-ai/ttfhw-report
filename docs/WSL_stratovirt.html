文档缺失 (4)

#1[高] Makefile yum-deps 使用 yum（已废弃），openEuler 24.03 LTS 中 yum 链接到 dnf，可正常工作但不够规范。 → 建议: 将 Makefile 中 yum-deps 目标改为 dnf-deps 或使用 dnf 命令。
#2[高] Cargo.lock 锁定 chacha20 v0.10.0（需要 edition2024），openEuler 24.03 官方 Rust 1.82.0 不支持，必须通过 rustup 安装更新版本的 Rust（>=1.85）。 → 建议: 在 README 或 docs/build_guide.md 中明确说明最低 Rust 版本要求（现阶段为 1.85+），或降级 chacha20 依赖到 0.9.x 以兼容 openEuler 官方 Rust 包。
#3[高] stratovirt 作为 hypervisor 依赖 KVM 内核模块，无法在容器和无 KVM 的环境中进行完整的集成测试和功能验证。 → 建议: 提供 mock KVM 接口或在 CI 中使用裸金属/嵌套虚拟化环境进行集成测试。
#4[低] cargo build 编译有 2 个 warning：ui crate 的 get_keyboard_state 函数未使用（dead_code），devices crate 的 MUT_ROOT_PORT 未使用导入（unused_imports）。 → 建议: 清理未使用的代码和导入（cargo fix --lib -p devices --tests）。

遇到的问题 (3)

openEuler 24.03 LTS 官方 Rust 包版本为 1.82.0，不支持 Cargo.lock 中 chacha20 v0.10.0 需要的 edition2024 特性。

方案: 通过 USTC 镜像安装 rustup，升级到 Rust 1.96.0（2026-05-25 版本）。

来源: Rust 版本不兼容

rustup.rs 官方站点无法直接访问，crates.io 下载速度极慢。

方案: rustup 使用 RUSTUP_DIST_SERVER=https://mirrors.ustc.edu.cn/rust-static 镜像安装；cargo 使用 USTC sparse registry 镜像（sparse+https://mirrors.ustc.edu.cn/crates.io-index/）。

来源: 网络限制

Docker 容器内 /dev/kvm 设备不存在，KVM 内核模块不可用，导致 13 个 balloon 集成测试失败，且 stratovirt 二进制无法运行。

方案: 这是容器环境的固有限制，非代码问题。单元测试（所有子 crate，~200+ 测试）全部通过，编译产物正确。如需完整集成测试，需要裸金属或 KVM 嵌套虚拟化环境。

来源: KVM 不可用（预期限制）
