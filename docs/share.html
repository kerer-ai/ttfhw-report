🔧 AI 行动

💡 AI 能区分"环境配置问题"和"代码问题"，优先选择无侵入式修复
Ascend/pytorch

20 核并发编译 OOM

🧠 AI 推理

dmesg 显示 cc1plus 被 OOM Killer 终止，内存耗尽。当前 ninja -j20 并发过高。C++ 模板实例化本身内存密集，大文件需降低并发。

🔧 AI 行动

export MAX_JOBS=4 && ninja -j4（降至 4 核并发）

结果：增量构建成功，内存峰值从 >16GB 降至 ~4GB

AI 也不是万能的

诚实展示 AI 的边界——哪些问题 AI 解决不了，为什么

硬件依赖（NPU）

UT 和样例需要 NPU 硬件，x86 环境不具备

AI 尝试：尝试安装 CANN 模拟器、寻找 CPU-only fallback

无法绕过— 物理硬件限制，非软件层面可解决。AI 正确标记为 not_run 而非 failed

容器网络/DNS 异常

Docker 容器内 DNS 解析失败，无法访问外部 pip/apt 源

AI 尝试：尝试配置 DNS、使用镜像源、修改 /etc/resolv.conf

部分可解决— 底层网络基础设施问题，AI 能力受限于容器环境权限

仓库源码 Bug

仓库自身代码存在编译错误（非环境问题）

AI 尝试：分析编译错误、尝试常见修复模式

不修改源码— 设计原则：AI 不修改被验证仓库的源码，确保验证结果客观可信

需交互式认证

私有依赖需要 SSH key 或 token 认证

AI 尝试：检查本地凭证、尝试匿名访问

标记为环境限制— 安全性约束：AI 不应处理认证凭证

带走什么

Takeaways

~3min
可复用方法论

不只是案例，更是可迁移的方法

以下 4 个核心原则可以应用到其他 AI 辅助开发场景

从文档出发，不猜测

AI 严格依据 README/构建脚本，不基于"常识"假设。这对保证验证结果客观性至关重要。

适用：任何需要理解外部代码的场景

不修改源码原则

AI 只能调整环境变量、构建参数、依赖版本——绝不修改仓库源码。这保证了"验证的是仓库本身"。

适用：代码审查、CI/CD 验证、兼容性测试

每步可追溯

AI 记录每步操作的决策依据、耗时、结果。不是黑盒——任何人都可以回溯验证过程。

适用：合规审计、质量追溯、知识沉淀

标准化输出

无论输入格式多混乱，AI 将结果归一化为 5 种标准状态。下游消费方（页面、飞书）无需理解原始格式。

适用：多源数据聚合、自动化报表、系统集成

TTFHW 真正在测什么：仓库的 Agent-Ready 程度

每次验证表面上是"构建能不能过"，实质上是在测试一个更深层的问题——这个仓库对 AI Agent 有多友好？一个 Agent 能否独立理解它、搭建它、跑通它？

文档清晰度

🤖 AI 能否从文档中自动提取构建命令、依赖列表和测试入口？

✅ Agent-Ready 标准

README 中明确标注构建命令（如 `bash ci/build.sh`）、依赖清单（requirements.txt / CMakeLists.txt）、测试入口（如 `pytest tests/`）

❌ 常见反模式

构建步骤隐藏在 CI 脚本深层、依赖分散在多个文件且未在 README 中索引、只有口头/飞书传递的"构建指南"

环境可复现性

🤖 AI 能否在无人指导的情况下复现构建和运行环境？

✅ Agent-Ready 标准

提供 Dockerfile / devcontainer.json / nix flake，声明系统级依赖和版本（如 cmake >= 3.20, gcc-11）

❌ 常见反模式

依赖宿主机预装的工具链、没有容器化方案、环境配置只在"老员工的机器上能跑"

依赖完整性

🤖 所有依赖是否显式声明，AI 能否一键安装？

✅ Agent-Ready 标准

Python: requirements.txt + setup.py；C++: CMakeLists.txt find_package；系统依赖: Dockerfile 中 apt/yum install；版本号固定而非 `latest`

❌ 常见反模式

代码中 import 了但未声明的包、私有源未配置访问方式、版本用 `latest` 导致不可复现

测试可执行性

🤖 测试入口是否标准化，AI 能否发现、运行并解析测试结果？

✅ Agent-Ready 标准

测试命令在 README 中明确标注、输出格式标准化（JUnit XML / JSON）、区分单元测试（无硬件依赖）和集成测试（需硬件）

❌ 常见反模式

测试需要特殊硬件但未声明、测试入口藏在 Makefile 深层 target、测试失败只输出堆栈无结构化报告

代码规范一致性

🤖 项目是否有自动化规范检查，AI 生成的代码能否自动符合项目风格？

✅ Agent-Ready 标准

配置 .pre-commit-config.yaml / .clang-format / .eslintrc / ruff.toml，CI 强制检查，pre-commit hook 自动修复格式问题

❌ 常见反模式

代码风格靠"人工 Review"维护、没有格式化配置、同一仓库内不同文件风格不一致导致 AI 代码风格飘忽

这 5 个维度构成了仓库的 Agent-Ready 评分卡。 59 个仓库的实际验证数据显示：维度覆盖越完整，AI 首次构建成功率越高，人工介入次数越少。

立刻就能做

让你的仓库进入 Agent-Ready 状态

来自 59 个仓库的实际验证数据——以下 5 步是影响最大的改进点，按优先级排列

Step 1

把"构建指令"写在 AI 能找到的地方

⚠️现状：9 个仓库的构建命令藏在 CI 脚本里，AI 需要扫描多层 YAML 才能找到
→做法：在 README 中用一个独立段落明确写出：`构建：bash ci/build.sh --release`
📈效果：AI 首次构建成功率从 ~60% 提升到 ~90%
Step 2

用 devcontainer 而不是"手动搭环境"

⚠️现状：15 个仓库缺少容器化配置，AI 需要自行选择基础镜像并安装系统依赖
→做法：在仓库根目录放置 .devcontainer/devcontainer.json，声明系统依赖和 VS Code 扩展
📈效果：环境准备时间从平均 12min 降至 <1min，且 100% 可复现
Step 3

锁定依赖版本，固定到具体版本号

⚠️现状：11 个仓库的依赖声明使用了 `latest` 或版本范围 `>=`，AI 在不同时间拉取到不同版本导致结果不一致
→做法：Python: pip freeze > requirements.txt 并提交；C++: CMakeLists.txt 中固定 find_package 版本；系统依赖: Dockerfile 中固定包版本
📈效果：消除了"昨天能跑今天不行"的依赖漂移问题
Step 4

把 pre-commit 配好，让 AI 帮你遵守规范

⚠️现状：18 个仓库没有 pre-commit 配置，AI 生成的代码风格无法自动校准
→做法：配置 .pre-commit-config.yaml：至少包含 trailing-whitespace、end-of-file-fixer、语言特定的 linter/formatter
📈效果：AI 生成代码的风格一致性从"靠运气"变成"靠机制"
Step 5

区分"无硬件测试"和"需硬件测试"

⚠️现状：22 个仓库的测试需要 NPU/GPU，但 README 未区分，AI 需要反复尝试才能判断
→做法：在 README 测试章节标注：`单元测试（无硬件依赖）：pytest tests/unit` vs `集成测试（需要 NPU）：pytest tests/integration --device npu`
📈效果：AI 不再浪费时间尝试无法执行的测试，验证时间缩短 ~40%

完成这 5 步之后，你的仓库就可以通过 /ttfhw-verify-pro 一键验证——AI 将独立完成构建全流程，你只需要审核结果。
